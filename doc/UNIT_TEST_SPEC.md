## 1. 目的

Mryl 言語インタープリタおよび C コードジェネレータに対して、以下の品質基準を満たすことを確認する。

- **C0（命令網羅）**: すべての実行可能な文が少なくとも 1 回実行されること
- **C1（分岐網羅）**: すべての条件分岐の真・偽パスが少なくとも 1 回実行されること
- **MC/DC（修正条件/判定網羅）**: 複合条件式（`&&`, `||`）において、各条件が独立して判定結果に影響することを確認すること

---

## 2. 境界値設計方針

各テスト観点は以下の値を網羅する：

| 観点 | 定義 |
|------|------|
| **最小値** | 型または仕様の定義範囲の下限値 |
| **最小値+1** | 下限値の直上値（下限境界の内側） |
| **仕様値** | 典型的な動作確認値（ゼロ、中間値など） |
| **最大値-1** | 上限値の直下値（上限境界の内側） |
| **最大値** | 型または仕様の定義範囲の上限値 |

> **注意**: 「最小値-1」「最大値+1」はオーバーフロー/未定義動作となるため、  
> Mryl 言語の型安全保証範囲外につき本仕様では実施対象外とする（→ §6 制約事項参照）

## 6. 実施制約事項（実施不可の理由）

### 6.1 型オーバーフロー境界（最小値-1、最大値+1）

**対象**: T13-N1, T13-N2, T13-N3, T13-N4  
**理由**: Mryl の型キャスト構文 `value(type)` では、型範囲外のリテラルを記述した場合に Python パーサが `int` として受け取り C 側で未定義動作を引き起こす。また、オーバーフロー時の動作は C 言語仕様上の未定義動作（signed integer overflow は UB）であり、テスト結果の再現性が保証されない。  
**エビデンス**: C 標準 §6.5 "If an exceptional condition occurs during the evaluation of an expression, the behavior is undefined."

### 6.2 i64 / u64 型の C ネイティブ出力

**対象**: T13-N5  
**理由**: 現在の C コードジェネレータ（`CodeGenerator.py`）は i64/u64 型変数の `println` 出力に `%d`（32 ビット整数フォーマット）を使用しており、64 ビット値が正しく表示されない。例: `9999999999` が `1410065407` として表示される（上位ビット切り捨て）。Python インタープリタでは正常動作する。  
**再現手順**: `tests/test_01_types.ml` の `i64=9999999999` の Native 出力が `1410065407` となることで確認済み。  
**対処方針**: `%lld` フォーマット指定子を使用するよう CodeGenerator を修正することで解決可能（別 Issue として管理）。

### 6.3 ゼロ除算

**対象**: T13-N6  
**理由**: Mryl 言語にはゼロ除算をキャッチする例外構文（`try`/`catch`）が存在しない。実行するとオペレーティングシステムの SIGFPE または未定義動作となり、プロセスが異常終了する。テストスイートの継続実行が不可能となるため実施対象外とする。

### 6.4 算術オーバーフロー（最大値+1 加算等）

**対象**: T13-N7  
**理由**: C 言語の符号付き整数オーバーフローは未定義動作（UB）であり、コンパイラ最適化によって結果が変わる。テスト結果の再現性が保証できないため実施対象外とする。なお、符号なし整数のオーバーフロー（ラップアラウンド）は C 言語で定義された動作だが、Mryl 言語に u32/u64 のオーバーフロー動作保証仕様がないため同様に対象外とする。

### 6.5 文字列長の境界値

**対象**: 文字列型の最大長境界  
**理由**: Mryl 言語仕様に文字列の最大長が定義されていない。メモリが許す限り可変長であり、上限境界値が存在しないため境界値分析の適用が不可能。

### 6.6 MC/DC — 3 条件以上の複合条件式

**対象**: `A && B && C` 形式  
**理由**: 本テスト実施時点で Mryl の標準テストコードに 3 条件以上の複合条件式が存在しない。MC/DC は 2 条件（&&, ||）に対してテストを実施する（test_14 参照）。

### 6.7 包含レンジ（`..=`）構文

**対象**: T15-08  
**理由**: `Parser.py` の `parse_range()` が `DOTDOT` トークンのみ処理しており、`..=`（包含上限）は `SyntaxError_` となる。`Parser.py` 修正で対応可能だが、本テスト時点では未実装のため実施対象外とする。代替として `0..(N+1)` による同等テストを実施済み。
